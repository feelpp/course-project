== Context

MagnetDB stores operational records as raw data files - time series of
50-70 physical channels recorded during magnet experiments. These
records need to be served to users through a web interface for
interactive visualization and analysis. This creates two distinct
challenges:

*The visualization challenge*: a record with 5000 data points and 70
columns is too large to send to a browser for plotting. The current
MagnetDB implementation uses naive modulo subsampling (keep every Nth
point), which discards peaks and distorts the visual representation.
Better algorithms exist - the goal is to implement and benchmark them.

*The lifetime challenge*: a magnet part may have operated across
100+ records over several years. To visualize trends (how did the
max stress evolve? is the resistance drifting?), these records must be
*stitched* into a coherent long-term series, handling gaps,
configuration changes, and varying sampling rates.

This project addresses both challenges by building a processing pipeline
for scientific time series data, with an emphasis on preserving the
physical fidelity of the signals.

== Scientific Objectives

[[obj-1-study-and-implement-downsampling-algorithms]]
=== 1. Study and implement downsampling algorithms

The core question: *given N data points, reduce to M much less than N points
while preserving the visual and statistical properties of the signal.*

Algorithms to implement and compare:

*Naive subsampling* (baseline): keep every floor(N/M)-th point. Simple but
loses peaks, valleys, and transients. This is what MagnetDB currently
does.

*Min-max bucketing*: divide the time axis into M/2 buckets, keep the min
and max from each bucket. Preserves extrema but doubles the number of
points per bucket and creates artificial oscillations.

*LTTB (Largest-Triangle-Three-Buckets)*: Steinarsson's algorithm (2013).
Divides data into M buckets and selects the point in each bucket that
maximizes the triangle area with the selected points in the adjacent
buckets. Has a geometric interpretation: it maximizes the visual
"importance" of each kept point. O(N) complexity.

*Douglas-Peucker* (Ramer-Douglas-Peucker): recursive simplification of
the polyline formed by the time series. Keeps points that deviate most
from the line connecting their neighbors. O(N log N) in best case, O(N^2)
in worst case.

*M4 aggregation*: for each bucket of data points, keep four values: min,
max, first, last. Designed specifically for time series visualization.
Produces 4M points but guarantees that the visual envelope is preserved.

For each algorithm, the student should:

* Implement it in Python (some have existing libraries; implement from
scratch for understanding, then validate against libraries)
* Analyze the computational complexity (theoretical and measured)
* Define appropriate error metrics
* Benchmark on real operational data

[[obj-2-define-error-metrics-for-scientific-data]]
=== 2. Define error metrics for scientific data

A key contribution of this project is defining what "faithful
downsampling" means for scientific time series. Standard image/graphics
metrics (e.g., Hausdorff distance between polylines) are not sufficient.
For magnet operational data, we care about:

*Peak preservation error*: abs(max(original) - max(downsampled)) /
max(original). For the magnetic field channel, losing the peak means
losing the most scientifically important datum.

*Integral preservation error*: abs(integral_original - integral_downsampled) /
abs(integral_original). For power or energy-related channels, the
integral represents total energy deposited.

*Frequency content preservation*: compare the power spectral density of
original and downsampled signals. Important for detecting oscillations
or instabilities.

*Statistical moment preservation*: compare mean, std, skewness, kurtosis
before and after downsampling.

*Visual fidelity*: overlay original and downsampled plots and compute
pixel-level difference (renders the time series as a rasterized image
and compares). This captures what the user actually sees.

[[obj-3-handle-multi-rate-and-irregular-time-series]]
=== 3. Handle multi-rate and irregular time series

Operational records may have:

* *Varying sampling rates*: different channels sampled at different
frequencies, or the sampling rate changes during a record (faster during
transients)
* *Gaps*: missing data points due to sensor dropout or acquisition
system issues
* *Timestamp irregularities*: jitter, duplicate timestamps,
non-monotonic timestamps

The student should implement:

* *Regularity detection*: given a time column, classify it as regular,
quasi-regular (with jitter), or irregular
* *Resampling*: align all channels to a common regular grid using
appropriate interpolation (linear for smooth channels, zero-order hold
for step-like channels such as coil currents)
* *Gap handling*: detect gaps (periods where Delta t exceeds a threshold),
mark them explicitly, and decide whether to interpolate or leave as NaN

[[obj-4-cross-record-stitching]]
=== 4. Cross-record stitching

Given a sequence of records for the same entity (site, magnet, or part),
build a coherent long-term series:

* *Temporal alignment*: records have absolute timestamps (Date +
Time columns). Stitch them chronologically with explicit gaps between
records.
* *Per-record aggregation*: for each record, extract a summary value
(max, mean, or a more complex statistic from Project A) to produce a
"one point per record" lifetime series.
* *Full stitching*: concatenate the full time series from all records
into a single series with gap markers. This is useful for detecting slow
drifts (resistance aging) that span multiple records.
* *Change-point detection*: identify records where the magnet
configuration changed (e.g., a helix was replaced). This can be detected
from discontinuities in baseline resistance or from metadata.

[[obj-5-anomaly-detection-exploratory]]
=== 5. Anomaly detection (exploratory)

With the processed time series, apply basic statistical process control
methods:

* *CUSUM (Cumulative Sum)*: detect persistent shifts in the mean of a
channel
* *EWMA (Exponentially Weighted Moving Average)*: detect gradual drifts
* *Threshold-based alerts*: flag records where a channel exceeds N
standard deviations from the historical mean

This objective is exploratory and secondary to objectives 1-4.

== Mathematical Content

* *Approximation theory*: piecewise-linear approximation of functions,
error bounds, optimal point selection
* *Computational geometry*: triangle area maximization (LTTB), polyline
simplification (Douglas-Peucker), convex hull concepts
* *Signal processing*: resampling theory, Nyquist criterion for
downsampling scientific data, anti-aliasing, spectral analysis (PSD
estimation via Welch's method)
* *Interpolation*: linear, cubic spline, zero-order hold; choice of
interpolation method based on signal physics
* *Algorithm analysis*: time and space complexity, benchmarking
methodology, statistical significance of benchmark results
* *Time series analysis*: change-point detection (CUSUM, PELT
algorithm), trend estimation, stationarity tests
* *Statistical process control*: CUSUM charts, EWMA charts, control
limits

== Evaluation Criteria

* *Algorithm understanding*: Can the student explain the geometric or
mathematical basis of each algorithm? Are the implementations correct?
* *Benchmark rigor*: Are the error metrics well-defined and physically
justified? Are benchmarks statistically sound (multiple runs, confidence
intervals)?
* *Practical recommendations*: Based on benchmarks, which algorithm
should MagnetDB use, with what parameters, for which types of channels?
* *Pipeline robustness*: Does `process_record()` handle edge cases
(empty columns, irregular timestamps, short records)?
* *Code quality*: Well-structured module, tests, documentation
* *Report quality*: Clear algorithm descriptions, insightful benchmark
analysis, actionable conclusions
